## 1. 项目概述

### 1.1 需求背景

随着银行信贷业务的多元化发展，对公、同业信贷场景的复杂度持续提升，且人行、银保监会对统一授信、集中度风险、限额管控的监管要求日趋严格。传统信贷额度管理系统存在以下痛点：

- 额度层级模糊
- 细项扩展困难
- 多业务映射硬编码
- 高并发响应慢

这些问题无法满足银行精细化管控、快速业务迭代和监管合规的双重需求。

### 1.2 核心目标

本平台旨在建设一套以额度管控为核心的信贷额度管控平台，实现以下目标：

#### 业务目标

- 满足人行/银保监会统一授信监管要求，实现单户/集团/关联方限额、集中度风险等核心指标的自动管控
- 构建集团→客户→批复→细项→合同的全层级额度管控体系，实现底层用信对上层额度的穿透式占用与扣减
- 支持对公、同业客户的精细化额度细项管理，实现不同细项对集团限额的差异化占用规则
- 实现业务品种与授信产品的多对多配置化映射，支持单业务合同跨多批复、多细项的额度多占，满足复杂业务场景
- 所有额度类型/细项采用配置化设计，新增业务无需修改表结构和代码，仅通过配置完成扩展

#### 技术目标

- **高性能**：实现毫秒级额度查询（≤100ms），支持10万+QPS高并发请求，满足业务实时管控需求
- **高可用**：平台整体可用性达99.9%，核心服务故障秒级切换，无单点故障
- **可扩展**：采用组件化、微服务架构，支持服务弹性扩缩容，业务组件可按需组合复用
- **数据一致性**：通过分布式锁、分布式事务保障高并发下额度数据的准确性，杜绝超占、漏占
- **可观测**：实现全链路监控、额度操作全审计、风险指标实时预警，内嵌监管合规校验逻辑

### 1.3 设计原则

#### 监管合规优先

所有设计均嵌入人行、银保监会统一授信、集中度控制等监管规则，做到合规内嵌、自动校验。

#### 组件化复用

核心功能与限额管理解耦为独立组件，接口标准化，支持按业务场景灵活组合，快速迭代新业务。

#### 配置化扩展

额度类型、细项规则、业务映射关系全部通过配置表管理，无硬编码，实现业务规则"零开发"调整。

#### 层级化管控

额度体系按"集团-客户-批复-细项-合同"分层，每层权责清晰，底层操作向上穿透，实现全链路额度可控。

#### 高性能实时

采用内存缓存、分布式计算、异步处理等技术，保障额度查询、占用、释放的实时性。

#### 数据全留痕

所有额度操作、调整、占用、释放均记录详细日志，支持全生命周期审计与追溯。

## 2. 监管合规要求

### 2.1 人行监管要求

- 单户授信余额 ≤ 净资本10%
- 集团客户授信余额 ≤ 净资本15%
- 关联方授信余额 ≤ 净资本20%

### 2.2 银保监会监管要求

- 实现集中度风险、资本充足率、流动性风险的实时监控与管控

### 2.3 合规内嵌机制

所有监管规则内置至系统核心逻辑，用信、调额等操作前自动完成合规校验，违规操作直接拦截。

### 2.4 客户范围界定

- 单一客户额度仅管理对公客户、同业客户，排除个人客户
- 对公客户额度细项：一般敞口额度、债券包销额度、主动投资额度，均占用集团敞口限额
- 同业客户额度细项：R1-R4类型额度占用集团敞口限额，R5类型额度占用集团低风险限额
- 所有额度层级的子分类均采用子表设计，禁止使用固定字段，新增细项仅需在子表添加记录，支持无限扩展

## 3. 总体架构设计

### 3.1 技术架构

本平台采用微服务架构，按"前端展示层-API网关层-微服务层-数据存储层-基础设施层"五层设计，核心功能与限额管理解耦为独立组件，支持按需组合、弹性扩展。

```
┌─────────────────────────────────────────────────────────────┐
│                      前端展示层                              │
│  Web前端 / 移动端 / 第三方系统集成                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      API网关层                               │
│  统一鉴权 / 路由转发 / 限流熔断 / 日志记录                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      微服务层                                │
│  额度查询服务 / 额度控制服务 / 风险预警服务 / 审计服务              │
│  额度计算服务 / 映射配置服务 / 系统管理服务                        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      数据存储层                             │
│  MySQL集群 / Redis集群 / RabbitMQ集群 / ES集群             │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      基础设施层                             │
│  Docker / Kubernetes / 监控告警 / 日志收集                   │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 业务架构

业务架构围绕六级额度管控体系设计，每层通过服务接口实现数据交互，底层操作向上穿透，实现全链路额度可控。

#### 核心业务链路

1. **集团层**：配置集团总限额、敞口/低风险等细项限额，设置占用规则
2. **客户层**：为对公/同业客户分配额度，客户细项额度受集团细项限额管控
3. **批复层**：根据客户需求进行业务授信批复，批复额度受客户细项额度管控
4. **细项层**：将批复额度拆分为具体授信产品细项，细项额度受批复额度管控
5. **映射层**：通过配置表将业务品种与授信产品细项建立多对多关联
6. **合同层**：用信业务合同根据映射关系，占用对应授信产品细项额度，细项额度实时扣减，向上穿透至批复、客户、集团层

### 3.3 组件化设计

平台采用组件化设计，将核心功能与限额管理解耦为独立组件，所有组件接口标准化，无耦合，支持按业务场景灵活组合，实现"组件复用、快速迭代"。

#### 核心功能组件

| 组件名称 | 核心职责 | 核心功能 | 技术特性 |
|---------|---------|---------|---------|
| 额度查询组件 | 毫秒级额度数据查询 | 多维度额度查询、额度充足性校验、额度状态统计、缓存预热 | Redis缓存、异步预加载、缓存穿透防护 |
| 额度控制组件 | 额度全生命周期操作管控 | 额度锁定/释放、占用/归还、超额校验、多占规则计算 | Redisson分布式锁、幂等性设计、原子操作 |
| 风险预警组件 | 额度风险实时监控与预警 | 阈值预警、集中度预警、趋势预警、合规预警、多渠道通知 | 流式计算、规则引擎、WebSocket实时推送 |
| 额度计算组件 | 复杂额度计算与分配 | 集团/客户额度计算、批复额度拆分、细项额度分配、多占方案最优计算 | 规则引擎、并行计算、配置化公式 |
| 审计追踪组件 | 额度操作全生命周期留痕 | 操作日志记录、额度变更追溯、审计报表生成、异常操作检测 | 异步日志收集、数据不可篡改、高效检索 |
| 监控大屏组件 | 额度指标实时展示与分析 | 全层级额度使用率、风险指标、预警信息、业务统计、自定义报表 | WebSocket实时推送、ECharts可视化、多维度钻取 |

#### 业务限额组件

| 组件名称 | 核心职责 | 核心功能 | 适用场景 |
|---------|---------|---------|---------|
| 基础限额组件 | 通用限额基础能力 | 限额定义、限额状态管理、限额生命周期控制、基础占用规则 | 所有限额类型的基础能力支撑 |
| 集团限额组件 | 集团层级限额管控 | 集团总限额配置、细项限额拆分、成员单位额度分配、集团集中度控制 | 集团客户额度管理 |
| 实时控制组件 | 用信环节实时限额校验 | 用信前额度校验、用信中额度锁定、用信后额度扣减、超额拦截 | 所有用信业务的实时管控 |
| 动态调整组件 | 额度全场景调整管理 | 自动调整、手动调整、临时调整、额度回收、调整审批流程 | 客户风险变化、业务需求变更 |
| 阈值预警组件 | 限额阈值精细化预警 | 多维度阈值配置、预警等级划分、预警触发规则、预警闭环处理 | 所有限额类型的阈值监控 |

#### 典型业务场景组件组合

1. **集团客户额度管控**：集团限额组件 + 实时控制组件 + 阈值预警组件 + 监控大屏组件
2. **对公客户基础用信管控**：基础限额组件 + 额度查询组件 + 额度控制组件 + 审计追踪组件
3. **同业客户多细项管控**：集团限额组件 + 实时控制组件 + 风险预警组件 + 额度计算组件
4. **复杂业务多批复多占**：额度计算组件 + 额度控制组件 + 映射配置服务 + 审计追踪组件

## 4. 核心额度体系设计

### 4.1 六级额度管控体系

本平台的额度体系为六级分层架构，从顶层集团限额到底层业务合同占用，每层均有明确的额度范围、管控规则和占用关系，底层操作向上穿透，实时扣减上层对应额度，所有细项均采用子表设计，支持配置化扩展。

```
第一层：集团限额
    ↓
第二层：集团限额细项
    ↓
第三层：单一客户额度
    ↓
第四层：客户额度细项
    ↓
第五层：业务授信批复/细项
    ↓
第六层：业务合同占用
```

### 4.2 额度层级详解

#### 第一层：集团限额

**核心定义**：银行对所有集团客户的最高总授信限额，是整个额度体系的顶层天花板，由银行根据净资本、监管要求统一配置。

**核心内容**：集团总授信限额，唯一且不可突破，受监管指标（如集团授信≤净资本15%）直接管控。

**管控规则**：
- 集团总限额 = 各集团细项限额之和
- 集团细项限额占用集团总限额，不可超占

#### 第二层：集团限额细项

**核心定义**：将集团总限额按风险类型拆分为多个细项限额，是集团层额度管控的核心，支持配置化扩展。

**核心细项**：
- 敞口限额
- 低风险限额（基础必配）
- 产品专用限额（可根据业务需求配置）
- 可扩展限额等

**管控规则**：
- 各细项限额之和 ≤ 集团总限额
- 对公客户所有细项额度仅占用集团敞口限额
- 同业客户R1-R4额度占用集团敞口限额，R5额度占用集团低风险限额
- 细项限额采用子表设计，新增细项仅需在group_quota_sub表添加记录

#### 第三层：单一客户额度

**核心定义**：银行对单个对公/同业客户的最高授信限额，是客户所有细项额度的总和，受集团细项限额管控。

**客户范围**：仅包含对公客户、同业客户，按客户ID唯一标识。

**管控规则**：
- 对公客户总额度 ≤ 集团敞口限额中分配给该客户的额度
- 同业客户总额度 = R1-R4额度 + R5额度
  - R1-R4额度 ≤ 集团敞口限额分配额
  - R5额度 ≤ 集团低风险限额分配额
- 客户总额度 = 客户各细项额度之和，不可超占

#### 第四层：客户额度细项

**核心定义**：将单一客户总额度按业务类型拆分为多个细项额度，是客户层额度管控的核心，支持配置化扩展。

**核心细项**：

**对公客户**：
- 一般敞口额度
- 债券包销额度
- 主动投资额度（可扩展）

**同业客户**：
- R1、R2、R3、R4、R5额度（按风险等级划分，可扩展）

**管控规则**：
- 对公客户所有细项额度之和 ≤ 客户总额度，且全部占用集团敞口限额
- 同业客户R1-R4额度之和 ≤ 客户总额度中的敞口部分，R5额度 ≤ 客户总额度中的低风险部分
- 客户细项额度采用子表设计，新增对公细项仅需在customer_quota_sub表添加记录

#### 第五层：业务授信批复/细项

**核心定义**：根据客户的具体业务需求，在客户细项额度范围内进行的业务授信批复，是额度从客户层到业务层的落地，分为批复层和细项层。

**批复层**：按业务类型进行的授信批复，如一般授信批复、专项授信批复、保函授信批复，批复额度 ≤ 对应客户细项额度。

**细项层**：将批复额度拆分为具体授信产品细项，如流动资金贷款、投标保函、履约保函，是业务用信的直接管控单位。

**管控规则**：
- 批复额度 ≤ 对应客户细项额度
- 多个批复额度之和 ≤ 同一客户细项额度
- 授信产品细项额度之和 ≤ 对应批复额度
- 细项额度采用子表设计，新增产品仅需在approval_quota_sub表添加记录
- 批复/细项额度状态与客户额度联动，客户额度冻结/回收时，对应的批复/细项额度同步冻结/回收

#### 第六层：业务合同占用

**核心定义**：客户实际用信的业务合同，是额度体系的最底层，合同额度直接占用授信产品细项额度。

**核心内容**：具体的信贷业务合同，如流动资金贷款合同、投标保函合同、同业存放合同，合同额度为实际用信金额。

**管控规则**：
- 合同额度 ≤ 对应授信产品细项的可用额度
- 多个合同额度之和 ≤ 同一细项额度
- 合同额度占用实时扣减授信产品细项额度，并向上穿透扣减批复额度、客户细项额度、集团细项额度
- 合同还款/终止时，释放的额度同步向上返还至各层级
- 审批中的合同临时锁定额度，审批拒绝/撤回时自动释放锁定额度

### 4.3 额度管控规则

#### 实时性

底层合同的占用/释放操作实时同步至上层所有相关层级，额度数据秒级更新。

#### 不可突破

任一层级的额度均不可超占，底层操作若导致上层额度超占，则操作失败。

#### 联动性

上层额度的冻结、调整、回收，同步联动至所有底层相关额度，确保全链路可控。

#### 独立性

不同集团、不同客户、不同批复的额度相互独立，无交叉占用。

## 5. 核心功能模块设计

基于六级额度体系和组件化架构，平台设计八大核心功能模块，覆盖额度从定义、配置、查询、占用到调整、监控、审计、系统管理的全生命周期管理。

### 5.1 额度配置管理

#### 功能说明

实现各层级额度的配置、分配、调剂、状态管理，是额度管控的基础模块。

#### 核心功能

1. **额度类型配置**：配置集团、客户、批复各层级的额度类型，设置类型编码、名称、风险属性、占用规则
2. **额度细项配置**：配置各层级的额度细项，关联上级额度类型，设置细项编码、名称、额度上限、扩展属性
3. **客户额度分配**：为对公/同业客户分配总额度和细项额度，关联集团细项限额，设置分配比例、有效期
4. **集团额度调剂**：集团内成员单位之间的额度调剂，调整各成员单位的额度分配额，确保集团总限额不超占
5. **额度状态管理**：对各层级额度进行启用、冻结、解冻、失效操作，状态变更同步联动底层相关额度

#### 业务流程

```
额度类型配置 → 额度细项配置 → 集团限额分配 → 客户额度分配 → 额度状态启用 → 额度正常使用
```

### 5.2 额度实时管控

#### 功能说明

这是平台的核心执行模块，实现额度的实时查询、锁定、释放、占用，保障用信业务的合规开展，响应时间≤100ms。

#### 核心功能

1. **毫秒级额度查询**：支持按集团、客户、批复、细项、合同等多维度查询额度总额、已用额度、可用额度、冻结额度
2. **额度锁定/释放**：
   - 用信审批前临时锁定对应额度
   - 锁定有效期可配置
   - 审批拒绝/撤回/超时自动释放
   - 审批通过则转为正式占用
3. **额度占用/归还**：
   - 用信合同生效时正式占用额度
   - 合同还款/终止时归还额度
   - 支持部分占用/部分归还
4. **超额管控拦截**：所有额度操作前均进行多维度充足性校验，任一环节额度不足则直接拦截操作，返回异常信息
5. **多占规则执行**：根据业务品种与授信产品细项的映射关系，计算单合同跨多批复、多细项的最优占用方案，自动执行多占操作

#### 业务流程

```
用信申请 → 业务品种匹配 → 映射关系查询 → 多维度额度校验 → 额度锁定 → 审批流程
→ 审批通过（额度占用）/审批拒绝（额度释放） → 合同生效 → 额度实时扣减
```

### 5.3 额度调剂管理

#### 功能说明

实现集团内成员单位之间、客户内部不同细项之间的额度调剂，满足业务灵活需求。

#### 核心功能

1. **集团额度分配**：将集团细项限额分配给各成员单位/客户，设置分配额度、有效期、调剂权限
2. **客户额度调剂**：客户内部不同细项额度之间的调剂，如对公客户一般敞口额度向债券包销额度调剂，需满足调剂规则
3. **集团内部调剂**：集团内成员单位之间的额度调剂，由集团发起，调剂后各成员单位额度总额不变，集团总限额不超占
4. **调剂审批流程**：所有额度调剂均需走审批流程，根据调剂金额设置不同审批节点，审批通过后生效
5. **调剂记录追溯**：记录所有调剂操作的发起方、接收方、调剂金额、审批流程、生效时间，支持全追溯

#### 调剂规则

- 额度调剂仅可在同风险属性的细项之间进行，如敞口额度之间可调剂，敞口额度与低风险额度不可调剂
- 调剂后接收方的额度 ≤ 其上级额度的可用额度，发起方的额度 ≥ 0
- 调剂额度有有效期，有效期满后自动返还至发起方

### 5.4 额度动态调整

#### 功能说明

支持自动调整和人工调整两种方式，满足客户风险变化、业务需求变更的额度调整需求，所有调整均需走审批流程，且同步联动底层额度。

#### 核心功能

##### 自动额度调整

基于客户风险评级、征信状况、经营指标的变化，系统自动计算调整额度，触发预警或直接执行调整（需配置规则）。

**调升规则**：客户风险评级提升、经营指标改善，且调整后额度不超上级限额。

**调降/回收规则**：客户风险评级下降、出现逾期/不良、经营指标恶化，系统自动调降额度，严重时强制回收额度。

##### 人工额度调整

由业务人员发起，为客户调整总额度或细项额度，分为永久调整和临时调整。

**永久调整**：长期调整客户额度，需走高级别审批流程。

**临时调整**：为满足客户短期业务需求的额度调整，设置有效期，有效期满后自动恢复原额度。

##### 额度回收

对风险恶化的客户，强制回收部分或全部额度，回收后底层相关批复/细项额度同步回收，正在审批的用信申请直接驳回。

##### 调整审批流程

- 根据调整类型、调整金额设置分级审批节点
- 支持会签、转签
- 审批过程中可查看客户全维度数据

##### 调整影响评估

调整前自动评估对底层业务的影响，展示受影响的批复、细项、合同，为审批提供依据。

### 5.5 业务品种映射

#### 功能说明

这是平台的核心业务模块，实现业务品种与授信产品细项的多对多配置化映射，以及单合同跨多批复、多细项的额度多占。

#### 核心功能

1. **业务品种配置**：配置银行所有信贷业务品种，如流动资金贷款、投标保函、同业存放，设置品种编码、名称、业务类型
2. **映射关系配置**：建立业务品种与授信产品细项的多对多映射关系，设置映射优先级、占用比例、有效期，支持一键启用/禁用
3. **多占方案计算**：单合同用信时，系统根据映射关系、各批复/细项的可用额度，自动计算最优多占方案，优先占用优先级高、使用率低的细项额度
4. **多占操作执行**：根据多占方案，自动对多个批复/细项进行额度锁定、占用、释放，所有操作原子性执行，任一环节失败则整体回滚
5. **映射关系监控**：监控映射关系的使用情况，展示各映射关系对应的用信合同、占用金额，支持按多维度筛选、统计

#### 映射规则

- 映射关系采用配置表管理，无硬编码，新增/修改映射关系无需重启服务，实时生效
- 多占方案需满足所有被占用批复/细项的额度充足性，且总占用金额 = 合同金额
- 多占操作采用分布式事务，确保所有批复/细项的额度操作一致，杜绝部分占用、部分失败
- 合同还款时，按多占比例向各批复/细项返还额度

### 5.6 风险监控预警

#### 功能说明

实现各层级额度风险的实时监控、智能预警、闭环处理，支持多维度监控指标、自定义预警规则、多渠道预警通知。

#### 核心功能

1. **监控指标配置**：配置多维度风险监控指标，如额度使用率、集中度比例、逾期率、风险评级，设置指标计算方式、统计周期
2. **预警规则配置**：为各监控指标设置预警阈值、预警等级（绿/黄/红）、触发条件，支持按集团、客户、业务类型自定义规则
3. **实时监控大屏**：可视化展示全层级额度数据、监控指标、预警信息，支持多维度钻取（如集团→客户→合同），数据实时更新
4. **智能风险预警**：指标达到预警阈值时，系统自动触发预警，通过短信、邮件、企业微信、平台消息等多渠道通知相关人员
5. **预警闭环处理**：预警信息支持认领、处理、反馈，记录处理过程、处理结果，处理完成后自动解除预警，未及时处理的预警逐级升级
6. **集中度风险监控**：实时计算单户、集团、行业、区域的额度集中度，与监管要求对比，超标时立即触发红色预警，并拦截新的用信申请
7. **趋势分析**：对额度使用率、占用金额、预警数量等指标进行趋势分析，展示近7/30/90天变化曲线，支持预测未来趋势

#### 预警规则

- 额度使用率 ≥ 80%（黄色预警）、≥ 90%（红色预警）
- 集团/单户集中度 ≥ 监管阈值（红色预警）
- 客户风险评级下降（黄色/红色预警，根据下降等级）
- 额度超占、异常占用（红色预警，立即拦截）
- 预警未及时处理（逐级升级预警）

### 5.7 审计追踪

#### 功能说明

实现平台所有额度操作、业务操作、系统操作的全生命周期留痕，支持多维度查询、审计报表生成，满足银行内部审计和外部监管的要求。

#### 核心功能

1. **操作日志记录**：记录所有操作的操作用户、操作时间、操作类型、操作内容、操作前后数据，包括额度配置、分配、调整、锁定、释放、占用、用信申请、审批等
2. **日志分类存储**：按操作类型将日志分为业务操作日志、系统操作日志、审计日志，采用不同的存储策略，确保日志不丢失、不篡改
3. **多维度日志查询**：支持按操作用户、操作时间、客户ID、合同编号、操作类型等多维度筛选日志，支持模糊查询、高级查询
4. **操作轨迹追溯**：输入客户ID/合同编号/批复编号，可查看该对象的全生命周期操作轨迹，展示所有操作的时间、人员、内容、数据变化
5. **审计报表生成**：内置多种审计报表，如额度调整审计表、用信操作审计表、预警处理审计表，支持自定义报表，报表可导出为Excel/PDF
6. **异常操作检测**：系统自动检测异常操作，如高频操作、跨权限操作、大额额度调整，触发预警并记录异常日志

#### 审计规则

- 所有日志不可删除、不可修改，仅可新增，确保数据的真实性和完整性
- 日志保存期限 ≥ 5年，满足监管要求
- 操作用户与银行统一身份认证系统关联，确保操作可追溯到具体人员

### 5.8 系统管理

#### 功能说明

实现平台的基础系统管理，包括用户、角色、权限、参数、日志、监控等，保障平台的安全、稳定运行。

#### 核心功能

1. **用户与权限管理**：基于RBAC模型设计，配置用户、角色、权限，支持菜单权限、按钮权限、数据权限，确保最小权限原则
2. **系统参数配置**：配置平台的核心系统参数，如缓存有效期、分布式锁超时时间、预警通知频率，支持参数热更新
3. **菜单与页面配置**：配置平台的前端菜单、页面，支持自定义菜单名称、排序、展示权限
4. **系统监控**：监控平台的服务器状态、数据库状态、缓存状态、服务状态，展示CPU、内存、磁盘、网络等性能指标，指标超标时触发预警
5. **日志管理**：查看平台的系统日志、错误日志、访问日志，支持按时间、级别、模块筛选，支持日志下载
6. **容灾备份**：配置平台数据的自动备份策略，支持全量备份、增量备份，备份文件可异地存储，支持一键恢复

## 6. 数据模型设计

### 6.1 数据库设计原则

本平台的数据模型基于MySQL 8.0设计，采用**"主表+子表"的结构实现额度细项的扩展**，所有核心表均添加索引、注释，风险监控等大数据量表采用分区表优化，数据访问层基于MyBatis设计，支持动态SQL、批量操作、结果映射。

#### 设计原则

1. **主表+子表**：所有额度层级的细项均采用主表记录核心信息、子表记录细项信息，支持配置化扩展
2. **最小冗余**：核心数据仅存储一次，关联数据通过外键/关联ID关联，杜绝数据冗余
3. **索引优化**：核心查询字段、关联字段、筛选字段均添加索引，提升查询性能
4. **分区表优化**：操作日志、风险监控等大数据量表按时间分区，提升批量查询和删除性能
5. **计算列**：可用额度、额度使用率等可通过公式计算的字段，采用MySQL计算列设计，实时更新，无需业务层计算
6. **外键约束**：核心表之间建立外键约束，保障数据的参照完整性，级联操作仅用于细项表的删除

### 6.2 核心数据表

按业务类型将核心数据表分为基础信息类、额度核心类、映射配置类、占用记录类、风险监控类、系统日志类六大类。

- 唯一索引：id_number

## 7. 关键技术实现

本平台基于Spring Cloud微服务架构，结合Redis、Redisson、RabbitMQ等中间件，解决高性能、高并发、数据一致性、配置化扩展等核心技术问题。

### 7.1 多级缓存架构

采用**"本地缓存（Caffeine）+分布式缓存（Redis）"**的多级缓存架构，保障额度数据的毫秒级查询。

#### 本地缓存

- 缓存平台高频访问的静态数据（如业务品种、映射关系、系统参数）
- 缓存有效期1小时
- 减少Redis访问

#### 分布式缓存

- 缓存所有动态额度数据（如各层级额度总额、已用金额、可用金额）
- 缓存Key按层级_唯一ID设计（如group_1001、customer_2001）
- 缓存有效期5分钟
- 额度操作后实时更新缓存

#### 缓存预热

- 系统启动时自动预热热点客户、热点批复的额度数据至Redis
- 避免首次查询缓存穿透

#### 缓存穿透防护

- 采用布隆过滤器过滤不存在的客户ID、合同编号
- 避免恶意请求穿透缓存访问数据库

### 7.2 分布式计算与异步处理

#### 分布式计算

- 额度计算、集中度计算等复杂计算任务
- 通过Spring Cloud Task实现分布式计算
- 提升计算效率

#### 异步处理

- 非实时的业务操作（如额度调整日志记录、预警通知）
- 通过RabbitMQ实现异步处理
- 避免阻塞主流程，提升接口响应速度

#### 批量操作

- 批量额度调整、批量用信申请等操作
- 采用MyBatis批量操作接口
- 减少数据库交互次数，提升处理效率

### 7.3 分布式锁机制

高并发场景下，为避免同一额度被多个请求同时操作，采用Redisson分布式锁保障额度操作的原子性。

#### 锁粒度

- 按额度唯一ID设计锁粒度（如quota_lock_customer_2001）
- 避免粗粒度锁导致的性能瓶颈

#### 锁超时

- 设置锁超时时间（30秒）
- 避免死锁
- 同时开启Redisson的看门狗机制，对未执行完成的操作自动续期

#### 锁释放

- 采用try-with-resources语法
- 确保操作完成后锁自动释放
- 避免手动释放遗漏导致的死锁

### 7.4 分布式事务

跨服务、跨表的额度操作（如多占操作、额度调剂），采用Seata AT模式实现分布式事务，保障数据一致性。

#### 事务边界

- 以用信申请、额度调整、多占操作等业务为事务边界
- 确保整个业务流程的原子性

#### 自动回滚

- 事务执行过程中若出现异常，Seata自动回滚所有已执行的操作
- 恢复数据至操作前状态

#### 幂等性设计

- 所有接口均实现幂等性
- 通过请求唯一ID避免重复提交
- 确保同一请求多次执行的结果一致

### 7.5 数据一致性保障

#### 实时校验

- 所有额度操作前均进行多维度数据校验（额度充足性、状态有效性、权限合法性）
- 校验不通过则直接拦截

#### 定期对账

- 系统每天凌晨自动对各层级额度数据进行对账
- 对比数据库与缓存的额度数据
- 对比底层合同与上层细项的占用数据
- 发现不一致则自动修正并触发预警

### 7.6 配置化管理

所有业务规则均通过配置表管理，结合Spring Cloud Config实现配置的热更新，无需重启服务，实现业务规则"零开发"调整。

#### 配置中心

- 采用Spring Cloud Config作为配置中心
- 将平台的核心配置（如预警规则、缓存有效期）存储在配置中心
- 支持多环境配置

#### 配置热更新

- 通过Spring Cloud Bus实现配置的热更新
- 配置修改后一键推送至所有服务
- 实时生效

#### 规则引擎

- 额度计算、预警规则、多占方案等复杂业务规则
- 采用Drools规则引擎实现
- 规则通过配置表管理，支持可视化配置

### 7.7 MyBatis优化

#### 动态SQL

- 通过MyBatis动态SQL实现多条件查询、批量操作
- 提升代码复用性

#### 结果映射

- 通过MyBatis的@Results、@One、@Many注解实现复杂关联查询的结果映射
- 减少业务层数据处理

#### 二级缓存

- 开启MyBatis二级缓存
- 缓存高频查询的结果
- 减少数据库访问

#### 分页插件

- 采用PageHelper分页插件
- 实现高效的分页查询
- 支持多数据库适配

### 7.8 MySQL优化

#### 存储引擎

- 所有核心表均采用InnoDB存储引擎
- 支持事务、行级锁、外键约束

#### 字符集

- 采用utf8mb4字符集
- 支持emoji表情和特殊字符，避免乱码

#### 性能参数

- innodb_buffer_pool_size：设置为物理内存的50%-70%
- max_connections：设置为2000
- query_cache_size：开启查询缓存

#### 慢查询优化

- 开启MySQL慢查询日志
- 记录执行时间>2秒的SQL语句
- 定期分析并优化慢查询

### 7.9 WebSocket实时推送

平台监控大屏、预警通知等需要实时更新的功能，采用WebSocket实现服务端到客户端的实时数据推送。

#### 长连接维护

- WebSocket建立长连接后，通过心跳包维护连接状态
- 心跳间隔30秒
- 连接断开后自动重连

#### 数据推送

- 额度数据变化、预警信息触发时
- 服务端主动将数据推送至所有连接的客户端
- 实现实时更新

#### 分组推送

- 按用户角色、权限进行分组推送
- 确保不同用户看到的推送数据符合其权限范围

## 8. 部署架构设计

本平台采用容器化、集群化部署架构，基于Docker+Kubernetes实现服务的容器化部署、集群管理、弹性扩缩容，保障平台的高可用、高可扩展，同时支持多环境部署，满足开发、测试、生产的不同需求。

### 8.1 多环境部署

平台支持开发、测试、预生产、生产四个环境的独立部署，各环境隔离，配置独立，确保生产环境的安全、稳定。

| 环境 | 部署方式 | 资源配置 | 用途 |
|------|---------|---------|------|
| 开发环境 | Docker Compose | 低配 | 开发人员日常开发、调试 |
| 测试环境 | Kubernetes | 中配 | 功能测试、性能测试、压测 |
| 预生产环境 | Kubernetes | 准生产配置 | 上线前验证、灰度测试 |
| 生产环境 | Kubernetes | 高配集群 | 正式业务运行 |

### 8.2 高可用集群

#### MySQL集群

- 采用一主多从架构
- 主库负责写操作，从库负责读操作
- 主从同步采用半同步复制，确保数据一致性
- 开启主从切换机制，主库故障时自动切换至从库，实现秒级故障恢复

#### Redis集群

- 采用主从复制+哨兵模式
- 主库负责写操作，从库负责读操作
- 哨兵节点监控主从状态
- 主库故障时自动选举新主库，实现故障自动切换

#### RabbitMQ集群

- 采用镜像集群架构
- 所有队列均创建镜像
- 确保消息不丢失
- 集群节点故障时不影响业务运行

#### Elasticsearch集群

- 采用主节点+数据节点+协调节点架构
- 主节点负责集群管理
- 数据节点负责数据存储和检索
- 协调节点负责请求转发
- 提升集群的处理能力和可扩展性

### 8.3 弹性扩缩容

基于Kubernetes的HPA（水平Pod自动扩缩）功能，实现平台服务的弹性扩缩容，根据CPU使用率、内存使用率、QPS等指标自动调整Pod数量，满足业务高峰期的需求，降低业务低峰期的资源消耗。

#### 扩缩容指标

- CPU使用率 ≥ 70%、内存使用率 ≥ 75%、QPS ≥ 5000，触发扩容
- CPU使用率 ≤ 30%、内存使用率 ≤ 35%、QPS ≤ 1000，触发缩容

#### 扩缩容范围

- 每个服务的Pod数量设置最小数（2个）和最大数（20个）
- 确保服务的高可用和资源合理利用

#### 扩缩容速度

- 扩容时每次增加1个Pod
- 缩容时每次减少1个Pod
- 避免资源波动过大

## 9. 风险控制机制

平台内置多层级、全流程的风险控制机制，从操作层、业务层、技术层三个维度实现风险的全面管控，确保平台安全、稳定、合规运行。

### 9.1 操作层风险控制

#### 身份认证

- 采用认证系统


#### 权限控制

- 基于RBAC模型实现细粒度的权限控制
- 支持菜单权限、按钮权限、数据权限
- 杜绝越权操作

#### 操作审计

- 所有操作均记录详细日志
- 包括操作用户、操作时间、操作内容、操作前后数据
- 支持全生命周期追溯

#### 异常操作检测

- 系统自动检测高频操作、跨权限操作、大额额度调整等异常操作
- 触发预警

### 9.2 业务层风险控制

#### 监管合规校验

- 所有用信、调额等操作前自动完成合规校验
- 违规操作直接拦截

#### 额度充足性校验

- 所有额度操作前进行多维度充足性校验
- 任一环节额度不足则操作失败

#### 集中度风险控制

- 实时计算单户、集团、行业、区域的额度集中度
- 与监管要求对比，超标时立即触发红色预警
- 拦截新的用信申请

#### 风险预警机制

- 多维度监控指标实时监控
- 指标达到预警阈值时自动触发预警
- 多渠道通知相关人员
- 预警闭环处理

### 9.3 技术层风险控制

#### 分布式锁保障

- 高并发场景下采用Redisson分布式锁
- 保障额度操作的原子性
- 避免超占、漏占

#### 分布式事务保障

- 跨服务、跨表的额度操作采用Seata分布式事务
- 保障数据一致性
- 异常自动回滚

#### 数据一致性保障

- 实时校验 + 定期对账
- 发现不一致自动修正并触发预警

#### 高可用保障

- 核心服务采用集群部署，无单点故障
- 数据库主从同步，缓存主从备份
- 故障自动切换
