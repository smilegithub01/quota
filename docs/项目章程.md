## 1. 项目概述

### 1.1 项目背景

随着银行信贷业务的多元化发展，对公、同业信贷场景的复杂度持续提升，且中国人民银行、中国银保监会对统一授信、集中度风险、限额管控的监管要求日趋严格。传统信贷额度管理系统存在以下痛点：

- **额度层级模糊**：缺乏清晰的层级划分，无法实现穿透式管控
- **细项扩展困难**：新增业务细项需要修改表结构和代码，开发周期长
- **多业务映射硬编码**：业务品种与授信产品的映射关系硬编码，灵活性差
- **高并发响应慢**：系统性能无法满足实时管控需求，影响业务办理效率

这些问题无法满足银行精细化管控、快速业务迭代和监管合规的双重需求，亟需建设一套符合银行级标准的信贷额度管控平台。

### 1.2 项目目标

#### 1.2.1 业务目标

1. **满足监管合规要求**
   - 实现人行/银保监会统一授信监管要求
   - 实现单户/集团/关联方限额、集中度风险等核心指标的自动管控
   - 嵌入监管规则，做到合规内嵌、自动校验

2. **构建六级额度管控体系**
   - 实现"集团→客户→批复→细项→合同"的全层级额度管控体系
   - 实现底层用信对上层额度的穿透式占用与扣减
   - 支持对公、同业客户的精细化额度细项管理

3. **支持业务品种灵活映射**
   - 实现业务品种与授信产品的多对多配置化映射
   - 支持单业务合同跨多批复、多细项的额度多占
   - 满足复杂业务场景的额度管控需求

4. **实现配置化扩展能力**
   - 所有额度类型/细项采用配置化设计
   - 新增业务无需修改表结构和代码，仅通过配置完成扩展
   - 实现业务规则"零开发"调整

#### 1.2.2 技术目标

1. **高性能**
   - 实现毫秒级额度查询（≤100ms）
   - 支持10万+QPS高并发请求
   - 满足业务实时管控需求

2. **高可用**
   - 平台整体可用性达99.9%
   - 核心服务故障秒级切换
   - 无单点故障

3. **可扩展**
   - 采用组件化、微服务架构
   - 支持服务弹性扩缩容
   - 业务组件可按需组合复用

4. **数据一致性**
   - 通过分布式锁、分布式事务保障高并发下额度数据的准确性
   - 杜绝超占、漏占现象
   - 实现ACID合规的额度操作

5. **可观测**
   - 实现全链路监控
   - 额度操作全审计
   - 风险指标实时预警
   - 内嵌监管合规校验逻辑

### 1.3 项目范围

#### 1.3.1 项目包含范围

1. **额度生命周期管控**（核心功能）
   - 额度创建：根据授信批复创建额度，建立六级额度管控体系
   - 额度调整：支持临时调整、永久调整、批量调整
   - 额度使用：支持单占、多占、预占模式，实现穿透式扣减
   - 额度冻结/解冻：支持全额冻结、部分冻结、条件冻结
   - 额度释放：支持正常还款、提前还款、合同终止等场景
   - 额度终止：支持到期终止、主动终止、强制终止

2. **额度体系管理**
   - 六级额度层级定义与配置
   - 额度类型与细项的配置化管理
   - 额度分配、调剂功能

3. **额度实时管控**
   - 额度查询与充足性校验
   - 额度锁定、占用、释放操作
   - 六级递进校验机制

4. **业务品种映射**
   - 业务品种与授信产品的多对多映射配置
   - 映射关系的动态调整

5. **白名单管理**
   - 白名单申请与审批
   - 白名单校验与豁免规则
   - 白名单使用记录

6. **风险监控预警**（简化）
   - 额度使用率监控
   - 集中度风险预警
   - 监管指标校验

7. **审计追踪**
   - 额度操作全生命周期留痕
   - 审计日志记录与查询

8. **系统管理**
   - 用户权限管理
   - 系统配置管理

#### 1.3.2 项目不包含范围

1. 个人客户额度管理
2. 核心会计核算功能
3. 贷前审批流程
4. 贷后催收管理
5. 风险评级模型计算
6. 外部系统对接（除核心银行系统外）

### 1.4 项目约束

#### 1.4.1 技术约束

1. 必须遵循银行级系统开发标准（ISO 20022、PCI-DSS）
2. 必须实现监管合规要求（Basel III、AML/KYC、GDPR）
3. 必须使用Spring Boot框架，遵循DDD架构原则
4. 必须使用BigDecimal进行所有金额计算
5. 必须实现AES-256数据加密和TLS 1.3传输加密
6. 必须实现双录入记账原则

#### 1.4.2 业务约束

1. 单户授信余额 ≤ 净资本10%
2. 集团客户授信余额 ≤ 净资本15%
3. 关联方授信余额 ≤ 净资本20%
4. 对公客户额度细项仅占用集团敞口限额
5. 同业客户R1-R4额度占用集团敞口限额，R5额度占用集团低风险限额

#### 1.4.3 时间约束

项目总工期：6个月
- 需求分析与设计：1个月
- 开发与测试：3个月
- 部署与上线：1个月
- 试运行与优化：1个月

#### 1.4.4 资源约束

1. 开发团队：10人（架构师1人、后端开发5人、前端开发2人、测试2人）
2. 硬件资源：生产环境服务器20台，测试环境服务器10台
3. 预算：500万元

### 1.5 项目假设

1. 银行已具备完善的客户数据管理体系
2. 核心银行系统提供必要的客户、合同数据接口
3. 监管政策在项目周期内无重大变化
4. 项目团队成员具备Spring Boot和银行系统开发经验
5. 基础设施（网络、服务器）按期到位

### 8.1 术语表

|------|------|
| 集团限额 | 银行对所有集团客户的最高总授信限额 |

| 低风险限额 | 集团限额中用于低风险业务的部分 |
