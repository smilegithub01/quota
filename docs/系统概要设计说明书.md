## 1. 引言

### 1.1 编写目的

本文档详细描述银行级信贷额度管控平台的系统架构设计、技术选型、核心组件设计、数据架构设计、安全架构设计、性能架构设计、可靠性架构设计和接口架构设计，为系统详细设计、开发、测试和部署提供技术依据。

### 1.2 文档范围

本文档涵盖以下内容：
- 系统架构设计（技术架构、业务架构、部署架构）
- 技术选型（开发语言、框架、中间件、数据库等）
- 核心组件设计（API网关、服务注册与发现、配置中心等）
- 数据架构设计（数据分层、数据一致性、数据缓存、数据备份）
- 安全架构设计（认证授权、数据加密、审计日志、安全防护）
- 性能架构设计（缓存架构、异步处理、数据库优化、负载均衡）
- 可靠性架构设计（高可用、容错、灾备）
- 接口架构设计（RESTful API、接口版本管理、接口文档管理）

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 微服务架构 | 将应用拆分为一组小型服务，每个服务运行在自己的进程中，服务间通过轻量级通信机制进行交互 |
| 领域驱动设计（DDD） | 一种软件开发方法，通过将复杂领域分解为多个限界上下文来管理复杂性 |
| 六级额度管控体系 | 集团限额→集团细项限额→客户额度→客户细项额度→批复额度→合同占用的六级分层架构 |
| 穿透式管控 | 底层操作实时影响上层所有相关额度，实现全链路额度可控 |
| 分布式事务 | 涉及多个服务或数据库的事务，需要保证ACID特性 |
| 分布式锁 | 在分布式环境下保证同一时间只有一个服务能够访问共享资源 |
| 熔断降级 | 当服务出现故障时，自动切断服务调用，防止故障蔓延，并提供降级服务 |
| 限流 | 限制系统处理的请求数量，防止系统过载 |

## 2. 系统概述

### 2.1 系统目标

#### 2.1.1 业务目标

1. **满足监管合规要求**
   - 实现人行/银保监会统一授信监管要求
   - 实现单户/集团/关联方限额、集中度风险等核心指标的自动管控
   - 嵌入监管规则，做到合规内嵌、自动校验

2. **构建六级额度管控体系**
   - 实现"集团→客户→批复→细项→合同"的全层级额度管控体系
   - 实现底层用信对上层额度的穿透式占用与扣减
   - 支持对公、同业客户的精细化额度细项管理

3. **支持业务品种灵活映射**
   - 实现业务品种与授信产品的多对多配置化映射
   - 支持单业务合同跨多批复、多细项的额度多占
   - 满足复杂业务场景的额度管控需求

4. **实现配置化扩展能力**
   - 所有额度类型/细项采用配置化设计
   - 新增业务无需修改表结构和代码，仅通过配置完成扩展
   - 实现业务规则"零开发"调整

#### 2.1.2 技术目标

1. **高性能**
   - 实现毫秒级额度查询（≤100ms）
   - 支持10万+QPS高并发请求
   - 满足业务实时管控需求

2. **高可用**
   - 平台整体可用性达99.9%
   - 核心服务故障秒级切换
   - 无单点故障

3. **可扩展**
   - 采用组件化、微服务架构
   - 支持服务弹性扩缩容
   - 业务组件可按需组合复用

4. **数据一致性**
   - 通过分布式锁、分布式事务保障高并发下额度数据的准确性
   - 杜绝超占、漏占现象
   - 实现ACID合规的额度操作

5. **可观测**
   - 实现全链路监控
   - 额度操作全审计
   - 风险指标实时预警
   - 内嵌监管合规校验逻辑

### 2.2 系统范围

#### 2.2.1 系统包含范围

1. **额度生命周期管控**（核心功能）
   - 额度创建：根据授信批复创建额度，建立六级额度管控体系
   - 额度调整：支持临时调整、永久调整、批量调整
   - 额度使用：支持单占、多占、预占模式，实现穿透式扣减
   - 额度冻结/解冻：支持全额冻结、部分冻结、条件冻结
   - 额度释放：支持正常还款、提前还款、合同终止等场景
   - 额度终止：支持到期终止、主动终止、强制终止

2. **额度体系管理**
   - 六级额度层级定义与配置
   - 额度类型与细项的配置化管理
   - 额度分配、调剂功能

3. **额度实时管控**
   - 额度查询与充足性校验
   - 额度锁定、占用、释放操作
   - 多占规则计算与执行
   - 六级递进校验机制

4. **业务品种映射**
   - 业务品种与授信产品的多对多映射配置
   - 映射关系的动态调整

5. **白名单管理**
   - 白名单申请与审批
   - 白名单校验与豁免规则
   - 白名单使用记录

6. **风险监控预警**（简化）
   - 额度使用率监控
   - 集中度风险预警
   - 监管指标校验

7. **审计追踪**
   - 额度操作全生命周期留痕
   - 审计日志记录与查询

8. **系统管理**
   - 用户权限管理
   - 系统配置管理

#### 2.2.2 系统不包含范围

1. 个人客户额度管理
2. 核心会计核算功能
3. 贷前审批流程
4. 贷后催收管理
5. 风险评级模型计算
6. 外部系统对接（除核心银行系统外）

### 2.3 设计原则

#### 2.3.1 监管合规优先

所有设计均嵌入人行、银保监会统一授信、集中度控制等监管规则，做到合规内嵌、自动校验。

#### 2.3.2 组件化复用

核心功能与限额管理解耦为独立组件，接口标准化，支持按业务场景灵活组合，快速迭代新业务。

#### 2.3.3 配置化扩展

额度类型、细项规则、业务映射关系全部通过配置表管理，无硬编码，实现业务规则"零开发"调整。

#### 2.3.4 层级化管控

额度体系按"集团-客户-批复-细项-合同"分层，每层权责清晰，底层操作向上穿透，实现全链路额度可控。

#### 2.3.5 高性能实时

采用内存缓存、分布式计算、异步处理等技术，保障额度查询、占用、释放的实时性。

#### 2.3.6 数据全留痕

所有额度操作、调整、占用、释放均记录详细日志，支持全生命周期审计与追溯。

## 3. 系统架构设计

### 3.1 架构风格

本系统采用**微服务架构**，基于**领域驱动设计（DDD）**思想，按照业务领域划分服务边界，每个服务独立部署、独立扩展、独立演进。

#### 3.1.1 微服务架构优势

1. **独立部署**：每个服务可以独立部署，不影响其他服务
2. **独立扩展**：根据业务需求独立扩展服务实例
3. **技术异构**：不同服务可以使用不同的技术栈
4. **故障隔离**：单个服务故障不会影响整个系统
5. **团队自治**：不同团队可以独立开发和维护不同的服务

#### 3.1.2 DDD设计原则

1. **限界上下文**：按照业务领域划分服务边界
2. **聚合根**：每个服务有自己的聚合根，保证数据一致性
3. **领域事件**：通过领域事件实现服务间通信
4. **防腐层**：隔离外部系统，保护核心领域模型

### 3.2 技术架构

本系统采用分层架构设计，分为"前端展示层-API网关层-微服务层-数据存储层-基础设施层"五层。

```
┌─────────────────────────────────────────────────────────────┐
│                      前端展示层                              │
│  Web前端 / 移动端 / 第三方系统集成                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      API网关层                               │
│  统一鉴权 / 路由转发 / 限流熔断 / 日志记录                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      微服务层                               │
│  额度查询服务 / 额度控制服务 / 风险预警服务 / 审计服务      │
│  额度计算服务 / 映射配置服务 / 系统管理服务                 │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                      数据存储层                             │
│  MySQL集群 / Redis集群 / RabbitMQ集群 / ES集群             │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐

│  Docker / Kubernetes / 监控告警 / 日志收集                   │
```

#### 3.2.1 前端展示层

**职责**：提供用户交互界面，接收用户请求，展示系统数据。

**技术选型**：
- Web前端：Vue 3 + Element Plus
- 移动端：uni-app
- 第三方集成：RESTful API

**核心功能**：
- 额度管理界面
- 额度查询界面
- 风险监控界面
- 审计查询界面
- 系统管理界面

#### 3.2.2 API网关层

**职责**：统一入口，负责请求路由、负载均衡、认证授权、限流熔断、日志记录等。

**技术选型**：Spring Cloud Gateway

**核心功能**：
- 统一鉴权：JWT认证、RBAC授权
- 路由转发：根据请求路径转发到对应服务
- 限流熔断：防止系统过载，保护系统稳定性
- 日志记录：记录所有请求日志，支持审计追踪

#### 3.2.3 微服务层

**职责**：实现具体的业务逻辑，提供RESTful API接口。

**技术选型**：Spring Boot + Spring Cloud

**核心服务**：
- 额度查询服务
- 额度控制服务
- 风险预警服务
- 额度计算服务
- 审计追踪服务
- 监控大屏服务
- 基础限额服务
- 集团限额服务
- 实时控制服务
- 动态调整服务
- 阈值预警服务
- 映射配置服务
- 系统管理服务

#### 3.2.4 数据存储层

**职责**：存储和管理系统数据，提供数据访问接口。

**技术选型**：
- 关系型数据库：MySQL 8.0
- 缓存：Redis 7.0
- 消息队列：RabbitMQ 3.12
- 搜索引擎：Elasticsearch 8.0

**核心功能**：
- 数据持久化存储
- 数据缓存加速
- 消息队列解耦
- 数据检索查询

#### 3.2.5 基础设施层

**职责**：提供系统运行的基础设施支持。

**技术选型**：

- 容器编排：Kubernetes
- 监控告警：Prometheus + Grafana

**核心功能**：
- 容器化部署
- 服务编排管理
- 系统监控告警
- 日志收集分析

### 3.3 业务架构

业务架构围绕六级额度管控体系设计，以额度生命周期管控为核心，每层通过服务接口实现数据交互，底层操作向上穿透，实现全链路额度可控。

#### 3.3.1 额度生命周期管控架构

额度生命周期管控是系统核心，涵盖额度从创建到终止的完整流程：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        额度生命周期管控流程                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐      │
│  │ 额度创建 │ → │ 额度调整 │ → │ 额度使用 │ → │ 额度冻结 │ → │ 额度终止 │      │
│  │         │   │         │   │         │   │  /解冻  │   │         │      │
│  └─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘      │
│       │             │             │             │             │            │
│       ↓             ↓             ↓             ↓             ↓            │
│  ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐      │
│  │授信批复 │   │临时/永久│   │单占/多占│   │全额/部分│   │到期/主动│      │
│  │合规校验 │   │批量调整 │   │预占模式 │   │条件冻结 │   │强制终止 │      │
│  └─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘      │
│                                     │                                       │
│                                     ↓                                       │
│                              ┌─────────────┐                               │
│                              │  额度释放   │                               │
│                              │ (还款/终止) │                               │
│                              └─────────────┘                               │
│                                                                             │
│  状态流转：待启用(PENDING) → 正常(ACTIVE) ↔ 冻结(FROZEN) → 已终止(TERMINATED)│
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 3.3.2 核心业务链路

1. **集团层**：配置集团总限额、敞口/低风险等细项限额，设置占用规则
2. **客户层**：为对公/同业客户分配额度，客户细项额度受集团细项限额管控
3. **批复层**：根据客户需求进行业务授信批复，批复额度受客户细项额度管控
4. **细项层**：将批复额度拆分为具体授信产品细项，细项额度受批复额度管控
5. **映射层**：通过配置表将业务品种与授信产品细项建立多对多关联
6. **合同层**：用信业务合同根据映射关系，占用对应授信产品细项额度，细项额度实时扣减，向上穿透至批复、客户、集团层

#### 3.3.3 业务流程图

```
额度创建流程：
授信审批通过 → 合规性校验 → 额度数据初始化 → 层级关联建立 → 创建完成

额度使用流程：
用信申请 → 白名单校验 → 六级递进校验 → 额度锁定 → 额度占用 → 穿透扣减 → 记录明细

额度释放流程：
还款/终止事件 → 释放金额计算 → 额度释放 → 穿透返还 → 记录明细

额度调整流程：
调整申请 → 合规性校验 → 审批流程 → 调整执行 → 联动更新 → 调整完成

额度冻结/解冻流程：
冻结申请 → 原因审核 → 冻结执行 → 状态变更 → 联动处理 → 冻结完成
```

## 7. 安全架构设计

### 7.1 认证授权设计

#### 7.1.1 认证机制

**认证方式**：JWT（JSON Web Token）

**认证流程**：
1. 用户登录，提交用户名和密码
2. 服务器验证用户名和密码
3. 服务器生成JWT令牌
4. 服务器返回JWT令牌给客户端
5. 客户端存储JWT令牌
6. 客户端每次请求携带JWT令牌
7. 服务器验证JWT令牌

**JWT令牌结构**：
```json
{
  "header": {
    "alg": "HS256",
    "typ": "JWT"
  },
  "payload": {
    "sub": "user123",
    "name": "张三",
    "roles": ["ADMIN", "USER"],
    "exp": 1707561600,
    "iat": 1707475200
  },
  "signature": "..."
}
```

#### 7.1.2 授权机制

**授权方式**：RBAC（Role-Based Access Control）

**授权流程**：
1. 用户登录，获取JWT令牌
2. 用户访问资源，携带JWT令牌
3. 网关验证JWT令牌
4. 网关提取用户角色
5. 网关检查用户是否有访问权限
6. 有权限则放行，无权限则拒绝

**角色权限设计**：
```
ADMIN: 所有权限
QUOTA_ADMIN: 额度管理权限
QUOTA_QUERY: 额度查询权限
RISK_ADMIN: 风险管理权限
AUDIT_ADMIN: 审计管理权限
SYSTEM_ADMIN: 系统管理权限
```

```java
    private final RedissonClient redissonClient;
    
    @EventListener
    @Async("recalculateExecutor")
    public void onExchangeRateUpdated(ExchangeRateUpdatedEvent event) {
        RLock lock = redissonClient.getLock("recalculate:lock:" + event.getCurrency());
        try {
            if (lock.tryLock(10, 300, TimeUnit.SECONDS)) {
                RecalculateTask task = RecalculateTask.builder()

                    .triggerSource("EXCHANGE_RATE_UPDATE")

                    .status(TaskStatus.PENDING)
                    .build();
                taskService.submitTask(task);
            }
        } finally {

                lock.unlock();
            }
        }
    }
```

**审计日志记录**：
```java
@Aspect
@Component
@RequiredArgsConstructor
public class SyncAuditAspect {
    
    private final AuditLogRepository auditLogRepository;
    
    @Around("@annotation(SyncOperation)")
    public Object auditSyncOperation(ProceedingJoinPoint joinPoint) throws Throwable {
        long startTime = System.currentTimeMillis();
        SyncOperation annotation = getAnnotation(joinPoint);
        
        AuditLog auditLog = AuditLog.builder()
            .operationType(annotation.operation())
            .operator(SecurityContextHolder.getCurrentUsername())
            .operatorIp(RequestContextHolder.getClientIp())
            .requestId(TraceContext.getTraceId())
            .build();
        
        try {
            Object result = joinPoint.proceed();
            auditLog.setStatus("SUCCESS");
            return result;
        } catch (Exception e) {
            auditLog.setStatus("FAILED");
            auditLog.setErrorMessage(e.getMessage());
            throw e;
        } finally {
            auditLog.setDurationMs(System.currentTimeMillis() - startTime);
            auditLogRepository.save(auditLog);
        }
    }
}
```

### 11.1 RESTful API设计

#### 11.1.1 设计原则

1. **资源导向**
   - 使用名词表示资源
   - 使用复数形式
   - 使用层级结构

2. **HTTP方法**
   - GET：查询操作
   - POST：创建操作
   - PUT：更新操作
   - DELETE：删除操作

3. **状态码**
   - 200：成功
   - 201：创建成功
   - 400：请求错误
   - 401：未授权
   - 403：禁止访问
