## 1. 引言

### 1.1 编写目的

本文档详细描述银行级信贷额度管控平台的功能需求、非功能需求、数据需求和接口需求，为系统设计、开发、测试和验收提供依据。

### 1.2 文档范围

本文档涵盖以下内容：
- 业务需求和业务规则
- 功能需求详细描述
- 非功能需求（性能、安全、可靠性等）
- 数据需求和数据关系
- 系统接口需求
- 质量属性和验收标准

### 1.3 术语定义

| 术语 | 定义 |
|------|------|
| 集团限额 | 银行对所有集团客户的最高总授信限额，是整个额度体系的顶层天花板 |
| 集团细项限额 | 将集团总限额按风险类型拆分的细项限额，如敞口限额、低风险限额 |
| 客户额度 | 银行对单个对公/同业客户的最高授信限额 |
| 客户细项额度 | 将客户总额度按业务类型拆分的细项额度 |
| 批复额度 | 根据客户需求进行的业务授信批复 |
| 授信产品细项 | 将批复额度拆分为具体授信产品细项 |
| 合同占用 | 客户实际用信的业务合同占用额度 |
| 额度穿透 | 底层合同占用实时扣减上层所有相关额度 |
| 多占规则 | 单业务合同跨多批复、多细项的额度占用规则 |
| 配置化管理 | 通过配置表管理业务规则，无需修改代码 |

## 2. 项目概述

### 2.1 产品背景

随着银行信贷业务的多元化发展，对公、同业信贷场景的复杂度持续提升，且监管要求日趋严格。传统信贷额度管理系统存在额度层级模糊、细项扩展困难、多业务映射硬编码、高并发响应慢等问题，无法满足银行精细化管控、快速业务迭代和监管合规的双重需求。

### 2.2 产品目标

1. **满足监管合规**：实现人行/银保监会统一授信监管要求，自动管控单户/集团/关联方限额、集中度风险等核心指标
2. **六级额度管控**：构建"集团→客户→批复→细项→合同"的全层级额度管控体系，实现穿透式占用与扣减
3. **精细化细项管理**：支持对公、同业客户的精细化额度细项管理，实现差异化占用规则
4. **灵活业务映射**：实现业务品种与授信产品的多对多配置化映射，支持复杂业务场景
5. **配置化扩展**：所有额度类型/细项采用配置化设计，新增业务无需修改表结构和代码

### 2.3 用户特征

| 用户角色 | 职责 | 使用频率 | 技术水平 |
|---------|------|---------|---------|
| 额度管理员 | 额度配置、分配、调整 | 高 | 中等 |
| 业务经办 | 额度查询、占用申请 | 高 | 低 |
| 风险管理人员 | 风险监控、预警处理 | 中 | 中等 |
| 审计人员 | 审计日志查询、异常检测 | 低 | 中等 |
| 系统管理员 | 系统配置、用户管理 | 低 | 高 |

## 3. 业务需求

### 3.1 业务流程

#### 3.1.1 额度配置流程

```
1. 额度管理员登录系统
2. 配置集团总限额
3. 配置集团细项限额（敞口限额、低风险限额等）
4. 为集团成员分配额度
5. 为客户分配总额度和细项额度
6. 进行业务授信批复
7. 配置授信产品细项
8. 配置业务品种与授信产品的映射关系
9. 启用额度
```

#### 3.1.2 额度占用流程

```
1. 业务经办发起用信申请
2. 系统根据业务品种查找映射的授信产品细项
3. 系统校验额度充足性
4. 系统锁定额度
5. 业务审批通过后，系统扣减额度
6. 额度占用实时穿透至上层所有相关层级
7. 记录操作日志
```

#### 3.1.3 额度释放流程

```
1. 客户还款或合同终止
2. 系统识别释放事件
3. 系统释放合同占用的额度
4. 额度释放实时返还至上层所有相关层级
5. 记录操作日志
```

#### 3.1.4 额度调整流程

```
1. 额度管理员发起调整申请
2. 系统校验调整合规性
3. 调整审批流程
4. 审批通过后，系统执行调整
5. 调整联动至底层相关额度
6. 记录操作日志
```

### 3.2 业务规则

#### 3.2.1 额度层级规则

1. **第一层：集团限额**
   - 集团总限额 = 各集团细项限额之和
   - 集团细项限额占用集团总限额，不可超占

2. **第二层：集团细项限额**
   - 各细项限额之和 ≤ 集团总限额
   - 对公客户所有细项额度仅占用集团敞口限额
   - 同业客户R1-R4额度占用集团敞口限额，R5额度占用集团低风险限额

3. **第三层：单一客户额度**
   - 对公客户总额度 ≤ 集团敞口限额中分配给该客户的额度
   - 同业客户总额度 = R1-R4额度 + R5额度
   - 客户总额度 = 客户各细项额度之和，不可超占

4. **第四层：客户额度细项**
   - 对公客户所有细项额度之和 ≤ 客户总额度，且全部占用集团敞口限额
   - 同业客户R1-R4额度之和 ≤ 客户总额度中的敞口部分，R5额度 ≤ 客户总额度中的低风险部分

5. **第五层：业务授信批复/细项**
   - 批复额度 ≤ 对应客户细项额度
   - 多个批复额度之和 ≤ 同一客户细项额度
   - 授信产品细项额度之和 ≤ 对应批复额度

6. **第六层：业务合同占用**
   - 合同额度 ≤ 对应授信产品细项的可用额度
   - 多个合同额度之和 ≤ 同一细项额度
   - 合同额度占用实时扣减授信产品细项额度，并向上穿透

#### 3.2.2 监管合规规则

1. **人行监管要求**
   - 单户授信余额 ≤ 净资本10%
   - 集团客户授信余额 ≤ 净资本15%
   - 关联方授信余额 ≤ 净资本20%

2. **银保监会监管要求**
   - 实现集中度风险、资本充足率、流动性风险的实时监控与管控

3. **合规内嵌机制**
   - 所有监管规则内置至系统核心逻辑
   - 用信、调额等操作前自动完成合规校验
   - 违规操作直接拦截

#### 3.2.3 额度管控规则

1. **实时性**
   - 底层合同的占用/释放操作实时同步至上层所有相关层级
   - 额度数据秒级更新

2. **不可突破**
   - 任一层级的额度均不可超占
   - 底层操作若导致上层额度超占，则操作失败

3. **联动性**
   - 上层额度的冻结、调整、回收，同步联动至所有底层相关额度
   - 确保全链路可控

4. **独立性**
   - 不同集团、不同客户、不同批复的额度相互独立
   - 无交叉占用

### 3.3 业务约束

1. **客户范围约束**
   - 单一客户额度仅管理对公客户、同业客户
   - 排除个人客户

2. **细项扩展约束**
   - 所有额度层级的子分类均采用子表设计
   - 禁止使用固定字段
   - 新增细项仅需在子表添加记录，支持无限扩展

3. **金额计算约束**
   - 所有金额计算必须使用BigDecimal
   - 金额精度为4位小数
   - 舍入方式为银行家舍入（HALF_EVEN）

4. **数据一致性约束**
   - 所有额度操作必须保证ACID特性
   - 高并发场景下使用分布式锁
   - 实现幂等性设计

## 4. 功能需求

### 4.1 额度生命周期管控

额度生命周期管控是本系统的核心功能，实现额度从创建到终止的完整生命周期管理。额度生命周期包括六个关键阶段：创建、调整、使用、冻结、解冻、终止，每个阶段都有明确的业务规则和操作流程。

#### 4.1.1 额度创建

**需求描述**：根据授信审批结果，在系统中创建额度记录，建立六级额度管控体系的初始数据。

**业务流程**：
```
授信审批通过 → 额度创建申请 → 合规性校验 → 额度数据初始化 → 层级关联建立 → 创建完成
```

**功能点**：
1. 集团限额创建：根据集团授信批复，创建集团总限额及细项限额
2. 客户额度创建：根据客户授信批复，创建客户总额度及细项额度
3. 批复额度创建：根据授信批复文件，创建批复额度及授信产品细项
4. 额度关联建立：建立六级额度的层级关联关系

**输入参数**：
- 授信批复编号（必填）
- 客户ID（必填）
- 额度类型（必填：集团/客户/批复）
- 额度金额（必填，BigDecimal）
- 币种（必填，默认CNY）
- 生效日期（必填）
- 到期日期（必填）
- 担保信息（可选）
- 创建人（必填）

**输出参数**：
- 额度ID
- 额度编号
- 创建结果（成功/失败）
- 错误信息（失败时）

**业务规则**：
1. **合规性校验**：
   - 单户额度 ≤ 净资本10%
   - 集团额度 ≤ 净资本15%
   - 关联方额度 ≤ 净资本20%
2. **层级约束**：
   - 客户额度 ≤ 集团细项限额分配额
   - 批复额度 ≤ 客户细项额度
   - 细项额度之和 ≤ 上层额度
3. **数据完整性**：
   - 额度编号系统自动生成，格式：QT+年月日+6位序号
   - 初始状态为"待启用"
   - 已用额度初始为0，可用额度等于总额度
4. **审计要求**：
   - 记录创建操作日志
   - 关联授信批复文件
   - 保存创建原因和依据

#### 4.1.2 额度调整

**需求描述**：根据业务需求或风险变化，对已创建的额度进行增加或减少调整。

**调整类型**：
1. **临时调整**：为特殊业务需求提供临时额度增加，需设定有效期，到期自动恢复
2. **永久调整**：根据客户风险等级变化或业务需求，永久性调整额度
3. **批量调整**：按规则对多个客户或集团进行批量额度调整

**业务流程**：
```
调整申请 → 合规性校验 → 审批流程 → 调整执行 → 联动更新 → 调整完成
```

**功能点**：
1. 发起调整申请
2. 调整合规性校验
3. 调整审批流程
4. 执行额度调整
5. 联动更新底层额度
6. 记录调整历史

**输入参数**：
- 额度ID（必填）
- 调整类型（必填：临时/永久/批量）
- 调整金额（必填，BigDecimal，正数为增加，负数为减少）
- 调整原因（必填）
- 有效期（临时调整必填）
- 申请人（必填）

**输出参数**：
- 调整记录ID
- 调整后额度
- 操作结果（成功/失败）
- 错误信息（失败时）

**业务规则**：
1. **调整限制**：
   - 减少调整后额度 ≥ 已用额度
   - 增加调整后额度 ≤ 上层额度可用余额
   - 临时调整需设定有效期，最长不超过1年
2. **联动规则**：
   - 上层额度调整联动影响下层额度
   - 集团额度调整需重新分配成员单位额度
   - 客户额度调整需校验批复额度是否超限
3. **审批规则**：
   - 调整金额超过阈值需审批
   - 临时调整需审批
   - 批量调整需高级别审批
4. **审计要求**：
   - 记录调整前后额度
   - 记录调整原因和依据
   - 临时调整记录到期时间

#### 4.1.3 额度使用（占用）

**需求描述**：客户用信时，系统根据业务品种映射关系，占用相应额度，实现穿透式扣减。

**使用模式**：
1. **单占模式**：单笔业务占用单个授信产品细项额度
2. **多占模式**：单笔业务跨多个批复、多个细项占用额度
3. **预占模式**：业务审批前预占额度，审批通过后确认占用

**业务流程**：
```
用信申请 → 白名单校验 → 额度充足性校验 → 额度锁定 → 审批通过 → 额度占用 → 穿透扣减 → 记录明细
```

**功能点**：
1. 白名单校验（豁免规则判断）
2. 六级递进额度充足性校验
3. 额度锁定（防止并发超占）
4. 额度占用执行
5. 穿透式扣减上层额度
6. 记录使用明细

**输入参数**：
- 客户ID（必填）
- 业务品种（必填）
- 用信金额（必填，BigDecimal）
- 合同编号（必填）
- 占用模式（必填：单占/多占/预占）
- 多占明细（多占时必填）

**输出参数**：
- 占用记录ID
- 占用明细列表
- 操作结果（成功/失败）
- 错误信息（失败时）

**业务规则**：
1. **校验顺序**（六级递进校验）：
   - 第一级：集团限额校验（总额度、细项额度）
   - 第二级：客户额度校验（总额度、状态有效性）
   - 第三级：客户细项额度校验（细项额度、类型匹配）
   - 第四级：批复额度校验（批复额度、批复状态）
   - 第五级：授信产品细项校验（产品额度、产品状态）
   - 第六级：业务品种映射校验（映射关系、占用比例）
2. **穿透扣减规则**：
   - 合同占用 → 扣减授信产品细项额度
   - 授信产品细项 → 扣减批复额度
   - 批复额度 → 扣减客户细项额度
   - 客户细项额度 → 扣减客户总额度
   - 客户总额度 → 扣减集团细项限额
   - 集团细项限额 → 扣减集团总限额
3. **并发控制**：
   - 使用分布式锁保证原子性
   - 锁定超时时间300秒
   - 支持幂等性操作
4. **白名单豁免**：
   - 完全豁免：不进行额度校验
   - 部分豁免：只校验未豁免部分
   - 阈值豁免：金额在阈值内豁免

#### 4.1.4 额度冻结

**需求描述**：对额度进行冻结操作，冻结后额度不可用于新业务，但已有占用不受影响。

**冻结类型**：
1. **全额冻结**：冻结全部额度
2. **部分冻结**：冻结部分额度
3. **条件冻结**：满足特定条件时自动冻结

**业务流程**：
```
冻结申请 → 冻结原因审核 → 冻结执行 → 状态变更 → 联动处理 → 冻结完成
```

**功能点**：
1. 发起冻结申请
2. 冻结原因审核
3. 执行额度冻结
4. 状态变更通知
5. 联动冻结下层额度
6. 记录冻结日志

**输入参数**：
- 额度ID（必填）
- 冻结类型（必填：全额/部分/条件）
- 冻结金额（部分冻结必填，BigDecimal）
- 冻结原因（必填）
- 冻结条件（条件冻结必填）
- 操作人（必填）

**输出参数**：
- 冻结记录ID
- 冻结后状态
- 操作结果（成功/失败）
- 错误信息（失败时）

**业务规则**：
1. **冻结限制**：
   - 冻结金额 ≤ 可用额度
   - 冻结后可用额度 = 原可用额度 - 冻结金额
   - 已占用额度不受影响
2. **联动规则**：
   - 上层额度冻结自动冻结下层额度
   - 集团额度冻结联动冻结所有成员单位额度
   - 客户额度冻结联动冻结所有批复额度
3. **业务影响**：
   - 冻结额度不可用于新业务
   - 已有合同可正常还款释放
   - 冻结期间额度不计入可用额度
4. **触发条件**（条件冻结）：
   - 客户风险等级下降
   - 逾期超过阈值
   - 监管预警触发

#### 4.1.5 额度解冻

**需求描述**：解除额度冻结状态，恢复额度可用性。

**解冻类型**：
1. **手动解冻**：人工操作解冻
2. **自动解冻**：满足条件自动解冻
3. **部分解冻**：解除部分冻结金额

**业务流程**：
```
解冻申请 → 解冻条件校验 → 审批流程 → 解冻执行 → 状态变更 → 联动处理 → 解冻完成
```

**功能点**：
1. 发起解冻申请
2. 解冻条件校验
3. 解冻审批流程
4. 执行额度解冻
5. 状态变更通知
6. 联动解冻下层额度
7. 记录解冻日志

**输入参数**：
- 冻结记录ID（必填）
- 解冻金额（部分解冻必填，BigDecimal）
- 解冻原因（必填）
- 操作人（必填）

**输出参数**：
- 解冻记录ID
- 解冻后状态
- 操作结果（成功/失败）
- 错误信息（失败时）

**业务规则**：
1. **解冻条件**：
   - 冻结原因已消除
   - 相关风险已解除
   - 审批流程已完成
2. **联动规则**：
   - 上层额度解冻联动解冻下层额度
   - 部分解冻按比例恢复下层额度
3. **状态变更**：
   - 解冻后状态恢复为"正常"
   - 可用额度 = 原可用额度 + 解冻金额
4. **审计要求**：
   - 记录解冻原因和依据
   - 关联原冻结记录
   - 保存解冻审批链路

#### 4.1.6 额度释放

**需求描述**：客户还款或合同终止时，释放已占用的额度，实现穿透式返还。

**释放场景**：
1. **正常还款**：客户按期还款，释放对应额度
2. **提前还款**：客户提前还款，释放对应额度
3. **合同终止**：合同提前终止，释放全部占用额度
4. **逾期核销**：逾期贷款核销，释放对应额度

**业务流程**：
```
还款/终止事件 → 释放金额计算 → 释放校验 → 额度释放 → 穿透返还 → 记录明细 → 释放完成
```

**功能点**：
1. 接收还款/终止事件
2. 计算释放金额
3. 释放校验
4. 执行额度释放
5. 穿透式返还上层额度
6. 记录释放明细

**输入参数**：
- 合同编号（必填）
- 释放金额（必填，BigDecimal）
- 释放原因（必填：正常还款/提前还款/合同终止/逾期核销）
- 还款流水号（还款时必填）
- 操作人（必填）

**输出参数**：
- 释放记录ID
- 释放明细列表
- 操作结果（成功/失败）
- 错误信息（失败时）

**业务规则**：
1. **释放计算**：
   - 正常还款：释放金额 = 还款本金
   - 提前还款：释放金额 = 提前还款本金
   - 合同终止：释放金额 = 剩余占用金额
   - 逾期核销：释放金额 = 核销本金
2. **穿透返还规则**：
   - 合同占用释放 → 返还授信产品细项额度
   - 授信产品细项 → 返还批复额度
   - 批复额度 → 返还客户细项额度
   - 客户细项额度 → 返还客户总额度
   - 客户总额度 → 返还集团细项限额
   - 集团细项限额 → 返还集团总限额
3. **并发控制**：
   - 使用分布式锁保证原子性
   - 支持幂等性操作
4. **审计要求**：
   - 记录释放原因和依据
   - 关联还款流水或终止事件
   - 保存释放前后额度快照

#### 4.1.7 额度终止

**需求描述**：额度到期或业务结束时，终止额度，清理相关数据。

**终止类型**：
1. **到期终止**：额度到期自动终止
2. **主动终止**：人工操作终止额度
3. **强制终止**：风险事件触发强制终止

**业务流程**：
```
终止触发 → 终止校验 → 占用清理 → 额度终止 → 状态变更 → 数据归档 → 终止完成
```

**功能点**：
1. 触发终止事件
2. 终止条件校验
3. 清理未完成占用
4. 执行额度终止
5. 状态变更通知
6. 数据归档处理
7. 记录终止日志

**输入参数**：
- 额度ID（必填）
- 终止类型（必填：到期/主动/强制）
- 终止原因（必填）
- 操作人（必填）

**输出参数**：
- 终止记录ID
- 终止后状态
- 操作结果（成功/失败）
- 错误信息（失败时）

**业务规则**：
1. **终止条件**：
   - 到期终止：额度到期日期到达
   - 主动终止：业务需求或客户申请
   - 强制终止：重大风险事件触发
2. **终止前检查**：
   - 检查是否有未完成的占用
   - 检查是否有未处理的冻结
   - 检查是否有未完成的调整
3. **数据清理**：
   - 终止前需清理所有占用
   - 终止后额度状态变更为"已终止"
   - 终止后数据归档至历史表
4. **联动规则**：
   - 上层额度终止联动终止下层额度
   - 批复额度终止不影响客户额度
   - 客户额度终止需先终止所有批复额度

#### 4.1.8 额度生命周期状态机

**状态定义**：
| 状态 | 状态码 | 说明 | 可转换状态 |

| 待启用 | PENDING | 额度已创建，等待启用 | 启用、终止 |
